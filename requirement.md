## 1. 项目概述

**技能德州扑克**是一款基于浏览器的多人实时网页对战游戏。游戏以标准多人 Texas Hold'em 为基础，在保留完整下注流程与牌型判定规则的前提下，叠加「技能系统」，通过信息操控与资源干扰增加策略深度。

**核心特点：**
- 支持多人对局（2–6 人），每位玩家初始持有 1000 筹码，筹码归零即淘汰，最后存活者获胜
- 每手牌提供固定 8 点能量，用于在对局开始前选定的 2 个技能中做资源分配
- 游戏逻辑全部在服务端执行，前端仅负责渲染
- 完整用户体系：注册、登录、个人资料，数据通过 MongoDB 持久化
- 战绩统计、成就解锁、对局回放等丰富的 MongoDB 驱动功能

**设计动机（来自课程提案）：**  
传统在线扑克高度依赖运气与心理博弈，缺乏可操控的策略层。通过引入技能系统，玩家可以在每手牌中做出影响信息对等性与底池规模的主动决策，使游戏兼顾扑克玩家的熟悉感与策略游戏玩家的操控感。

## 2. 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **前端** | React + Tailwind CSS | 组件化 UI，纯渲染层，无游戏逻辑 |
| **后端框架** | Node.js + Express | RESTful API 服务，用户认证，战绩/成就接口 |
| **实时通信** | Socket.io | 游戏状态实时双向同步，技能事件推送 |
| **牌型判定** | pokersolver（npm） | 7 选 5 最优组合自动计算，无需自行实现 |
| **游戏状态** | 服务端内存（Room 对象） | 单局状态随对局生命周期存在于内存 |
| **数据库** | MongoDB（Mongoose） | 用户、战绩、成就、回放数据持久化 |
| **身份认证** | JWT（jsonwebtoken） | 无状态认证，Token 存于客户端 |
| **快速匹配队列** | Redis | 维护等待匹配队列，可选实现 |

## 3. 用户系统

### 3.1 注册与登录

**注册（POST /api/auth/register）**
- 字段：`username`（唯一，3-20 字符）、`email`（唯一）、`password`（≥8 字符，bcrypt 加密存储）
- 注册成功后返回 JWT Token
- 用户名重复、邮箱重复均返回明确错误信息

**登录（POST /api/auth/login）**
- 字段：`email` + `password`
- 验证通过返回 JWT Token（有效期 7 天）
- 密码错误/用户不存在均返回统一错误（不区分，防止枚举攻击）

**Token 刷新（POST /api/auth/refresh）**
- 使用 Refresh Token 换取新 Access Token

### 3.2 个人资料

**查看资料（GET /api/users/:userId）**
- 公开字段：`username`、`avatar`、`createdAt`、`totalWins`、`totalGames`、`achievements`（已解锁列表）

**修改资料（PUT /api/users/me）**
- 可修改：`username`（唯一检查）、`avatar`（头像 URL 或预设头像 ID）
- 需要 JWT 认证

**修改密码（PUT /api/users/me/password）**
- 需要提供旧密码验证

### 3.3 前端页面

- **注册页**：表单 + 实时校验
- **登录页**：表单 + 记住我（LocalStorage Token）
- **个人主页**：头像、用户名、胜率环形图、近期战绩列表、已解锁成就展示
- **导航栏**：登录后显示用户名、头像、退出按钮；未登录显示登录/注册入口

## 4. 房间与匹配

### 4.1 房间码邀请

- 登录用户可创建房间，服务端生成 6 位唯一房间码
- 其他用户输入房间码加入，支持 2–6 名玩家；房主可手动开始，或房间满员后自动进入技能选择阶段
- 房间状态存储于服务端内存（Room 对象），与 MongoDB 用户数据解耦
- 房间码在对局结束后失效

### 4.2 快速匹配

- 用户点击「快速匹配」，进入 Redis 等待队列
- 队列中存在等待玩家时立即配对并创建房间，两端同时收到 `matchFound` 事件
- 前端显示「匹配中…」等待 UI，含取消按钮（`cancelMatch` 事件）
- 若 Redis 不可用，降级为轮询内存队列

## 5. 技能选择

- **时机**：双方进入房间后、发牌前，展示技能选择界面
- **规则**：每位玩家从 **6 个技能**中选择 **2 个**（技能池可扩展，见第 10 章）
- **可见性**：双方选择完毕后，互相可见对方所选技能名称
- **约束**：每手牌每个技能最多使用 1 次；能量不足则无法使用（按钮置灰）
- **超时**：选择界面限时 30 秒，超时随机分配 2 个技能

## 6. 单手牌流程

每手牌按以下顺序执行（标准 Heads-Up 德州扑克流程）：

| 阶段 | 操作 | 技能窗口 |
|------|------|----------|
| **发牌** | 双方各得 2 张底牌；能量重置为 8；收取盲注 | 窗口 A（3 秒）：可使用信息型技能（线索嗅探） |
| **Pre-Flop 下注** | 标准德州下注流程；庄家先行动 | 可使用翻倍协议（独立触发） |
| **Flop** | 翻开 3 张公共牌 | 窗口 B（4 秒）：可使用换牌技能（孤注一掷）、线索嗅探 |
| **Flop 下注** | 标准德州下注流程；大盲位先行动 | 可使用翻倍协议 |
| **Turn** | 翻开第 4 张公共牌 | 窗口 C（4 秒）：可使用线索嗅探 |
| **Turn 下注** | 标准德州下注流程 | 可使用翻倍协议 |
| **River** | 翻开第 5 张公共牌 | 窗口 D（4 秒）：可使用线索嗅探；摊牌前最后时机 |
| **River 下注** | 标准德州下注流程 | 可使用翻倍协议 |
| **摊牌 & 结算** | pokersolver 7 选 5 比牌型；展示手牌评分；筹码结算；能量重置；写入回放记录 | — |

### 庄家规则
- 庄家按顺时针方向每手牌轮换；庄家左侧为小盲注（SB），再左侧为大盲注（BB）
- Pre-Flop：大盲注左侧玩家先行动，行动顺序顺时针进行；Flop 后：小盲注（或最左侧存活玩家）先行动
- 技能冲突时：行动顺序靠后的玩家拥有反应优先权

### 决策时限
- 每次行动（下注/弃牌/过牌）限时 **15 秒**，倒计时显示在操作区
- 超时自动执行：有注额时自动 fold，无注额时自动 check
- 技能窗口超时视为跳过本窗口（不使用技能）

## 7. 下注系统

遵循标准德州扑克下注规则，支持以下操作：

| 操作 | 触发条件 | 说明 |
|------|----------|------|
| **Check（过牌）** | 当前无人下注 | 维持现状，进入下一阶段或轮到对手 |
| **Bet（下注）** | 当前无人下注 | 通过滑块输入金额，最低为大盲注金额 |
| **Call（跟注）** | 对手已下注/加注 | 跟随当前最高注额 |
| **Raise（加注）** | 对手已下注 | 加注金额 ≥ 当前最高注额的 2 倍 |
| **Fold（弃牌）** | 任意时机 | 放弃本手牌，对手赢得底池 |
| **All-in** | 任意时机 | 押上全部剩余筹码 |

**盲注设定：**
- 小盲注：10 筹码；大盲注：20 筹码（固定，不随对局变化）

## 8. 牌型判定与结算

使用 `pokersolver` npm 包完成 7 选 5 最优组合自动计算。

### 8.1 牌型大小（由高到低）

| 级别 | 牌型 | 基础评分 | 说明 |
|------|------|----------|------|
| 1 | 皇家同花顺 | 30 | A-K-Q-J-10 同花色 |
| 2 | 同花顺 | 25 | 同花色连续五张（非 A 高） |
| 3 | 铁支（四条） | 22 | 四张相同点数 + 1 张杂牌 |
| 4 | 葫芦 | 18 | 三条 + 一对 |
| 5 | 同花 | 15 | 五张相同花色（不连续） |
| 6 | 顺子 | 14 | 五张连续点数（不同花色） |
| 7 | 三条 | 10 | 三张相同点数 + 2 张不同杂牌 |
| 8 | 二对 | 8 | 两组不同点数的对子 + 1 张杂牌 |
| 9 | 对子 | 5 | 一对相同点数 + 3 张杂牌 |
| 10 | 散牌（高牌） | 2 | 无法构成以上任意牌型 |

> 基础评分来源于 PDF 需求，用于摊牌时在 UI 上直观展示双方「手牌强度分」，不影响胜负判定（胜负仍由 pokersolver 比较结果决定）。

### 8.2 牌点加成（用于展示评分）

在基础评分基础上叠加牌点加成，使同牌型之间有数值区分（仅用于 UI 展示，不影响胜负）：

| 牌型 | 加成公式 |
|------|----------|
| 散牌 | 最大牌点数 ÷ 4（A=14，K=13，Q=12，J=11） |
| 对子 | 对子点数 ÷ 2 |
| 二对 | 较大对子点数 ÷ 2 |
| 三条 | 三条点数 ÷ 2 |
| 顺子 / 同花 | 最高牌点数 ÷ 3 |
| 葫芦 / 四条 / 同花顺 | 核心牌点数 ÷ 2 |

### 8.3 同牌型比较（踢脚牌规则，由 pokersolver 处理）

- **铁支**：比四条点数；相同则比杂牌（kicker）
- **葫芦**：先比三条点数，相同则比对子点数
- **同花 / 散牌**：从最大一张逐张向下比较
- **顺子 / 同花顺**：比最大一张；A 在 A-2-3-4-5 中作最小牌
- **三条**：比三条点数，相同依次比两张杂牌
- **二对**：先比大对，再比小对，最后比杂牌
- **对子**：比对子点数，相同依次比三张杂牌

### 8.4 A 的特殊规则

- 点数排序：A > K > Q > J > 10 > … > 2
- 顺子中 A 可作最大（A-K-Q-J-10）或最小（A-2-3-4-5）

### 8.5 筹码结算逻辑

- 胜者赢得本手底池全部筹码
- **平局（同大）**：筹码平分；双方各 +2 点能量（在本手能量重置前结算）
- 某玩家筹码归零时该玩家被淘汰；最后存活的玩家获胜，写入 MongoDB 战绩记录

## 9. 能量系统

| 条件 | 能量变化 |
|------|----------|
| 每手牌开始 | 固定重置为 **8 点** |
| 平局（同大） | 双方各 **+2 点**（重置前结算，不超过上限 8） |
| 使用技能 | 按各技能能量成本扣除 |
| 每手牌上限 | **8 点**（不可超过） |
| 能量不足 | 对应技能按钮置灰，不可点击 |

> 能量**不跨手积累**，每手牌结束后完全重置。固定能量机制让每手牌的技能决策更纯粹——在哪个时机花多少能量是核心策略选择。

## 10. 技能系统

对局开始前，每位玩家从 **6 个技能**中选择 **2 个**。技能池设计为可扩展数组，新增技能只需添加技能对象，不修改核心结算逻辑。

### 10.1 技能池（6 个）

| 技能名 | 类型 | 能量消耗 | 效果描述 |
|--------|------|----------|----------|
| **线索嗅探** | 主动 | 3 点 | 查看对手一张底牌（随机）。结果仅自己可见（定向推送）；对手仅知道「有一张底牌被查看」，不知是哪张 |
| **孤注一掷** | 主动 | 2 点 | 将自己一张随机底牌换成牌堆中的随机牌（可能更好也可能更差）。对手知道换牌发生，但不知换了哪张。换牌后自动重算最优 5 张 |
| **翻倍协议** | 主动 | 3 点 | 向对手发起本手底池翻倍提案。对手 5 秒内响应：接受则本手底池翻倍；超时或拒绝则**被提案方**损失 1 点能量，提案方能量不消耗 |
| **盾反** | 响应 | 2 点 | 在对手使用技能后立即响应（5 秒内）：免疫该技能效果并将其反弹给对手，对手能量照常消耗。**仅可反弹线索嗅探和孤注一掷**，不可反弹翻倍协议 |
| **精准换牌** | 主动 | 4 点 | 将自己一张**指定**底牌换成牌堆中的随机牌（可主动选择换哪张，比孤注一掷更可控）。对手知道换牌发生 |
| **信息封锁** | 主动 | 3 点 | 本手牌内，对手的线索嗅探和精准换牌无法对你生效（被动防护）。施放后在事件日志中对双方可见 |

### 10.2 技能可用时机

| 技能 | 可使用的时机 |
|------|-------------|
| **线索嗅探** | 任意技能窗口（发牌后窗口 A、Flop 窗口 B、Turn 窗口 C、River 窗口 D） |
| **孤注一掷** | 发牌后窗口 A、Flop 窗口 B（公共牌出来前换牌意义更大） |
| **精准换牌** | 发牌后窗口 A、Flop 窗口 B |
| **翻倍协议** | 任意下注阶段开始前（独立触发，不占技能窗口） |
| **信息封锁** | 任意技能窗口 |
| **盾反** | 对手使用线索嗅探、孤注一掷或精准换牌后，在同一窗口内 5 秒内响应 |

### 10.3 技能触发顺序与规则

- **顺序触发**：轮到谁行动谁可使用技能，服务端顺序处理，无并发冲突
- **每手限制**：每个技能每手牌最多使用 1 次
- **目标选择**：多人局中，使用需要指定目标的技能（如线索嗅探、精准换牌）时，需手动选择目标玩家
- **盾反确认**：对手技能生效前，服务端向盾反持有者推送确认弹窗（5 秒倒计时），超时视为放弃，盾反能量不消耗

## 12. 断线重连

- 玩家断线后，服务端保留其房间状态 **30 秒**
- 30 秒内重连：恢复完整游戏状态（`reconnectState` 事件），继续对局
- 超时（30 秒）：当前手牌自动 fold，本手底池由其余玩家正常结算；若仅剩一名玩家存活，对局结束并写入战绩
- 断线期间轮到断线方行动时，15 秒倒计时照常计时，超时自动执行

## 13. 战绩系统与统计面板

### 13.1 战绩记录

每局对局结束后，将对战结果写入 MongoDB：

- 胜者、败者（userId）
- 持续手牌数
- 双方选用的技能列表
- 双方使用技能的次数统计
- 对局结束时间戳
- 对局时长（秒）
- 最终筹码比（胜者/败者）

### 13.2 个人统计面板

**前端页面：个人主页 → 统计 Tab**

| 统计项 | 说明 |
|--------|------|
| 总对局数 | 参与的全部对局数量 |
| 胜率 | 胜场 / 总场次（百分比 + 环形图） |
| 平均对局时长 | 所有对局时长的平均值 |
| 最常用技能 TOP 3 | 技能使用频率柱状图 |
| 胜率最高技能 | 使用该技能时的胜率 |

### 14.2 解锁流程

1. 每局结束后服务端检查成就触发条件
2. 满足条件则写入用户成就记录（`unlockedAt` 时间戳）
3. 通过 Socket.io `achievementUnlocked` 事件通知前端展示解锁动画
4. 个人主页成就 Tab 展示已解锁与未解锁（未解锁显示为锁定状态）

## 15. 对局回放功能

### 15.1 数据采集

对局进行中，服务端将以下关键事件写入内存缓冲区，对局结束后批量写入 MongoDB：

- 每手牌的发牌结果（双方底牌、公共牌顺序）
- 每次下注操作（操作类型、金额、时间戳）
- 每次技能使用（技能名、使用方、时间戳、效果结果）
- 每手牌的摊牌结果（双方最终手牌、评分、筹码变化）
- 对局结束事件

### 15.2 回放查看

**前端页面：战绩列表 → 某场对局 → 查看回放**

- 展示本场对局的手牌数量概览
- 可按手牌序号查看：该手牌的双方底牌、公共牌翻牌顺序、技能使用时间线、下注记录
- 事件时间线以「事件日志」形式重现，与对局中体验一致
- 回放数据为只读，不支持真实游戏操作

### 15.3 存储策略

- 仅保留最近 **50 场**对局的详细回放数据（按 `endedAt` 排序，超出自动删除最旧记录）
- 回放数据以 BSON 嵌套文档方式存储于 `MatchReplay` 集合

| 类别 | 要求 |

## 21. 边界情况处理

| 边界情况 | 处理方式 |
|----------|----------|
| **平局（同大）** | 筹码平分；双方各 +2 点能量（本手重置前结算，不超过上限 8） |
| **翻倍协议超时/拒绝** | 提案方**不消耗**能量；被提案方损失 1 点能量 |
| **盾反超时** | 视为放弃；对手技能正常生效；盾反能量**不消耗** |
| **盾反 vs 翻倍协议** | 翻倍协议为提案型，盾反无法反弹 |
| **双方都选了盾反** | 先触发技能的一方被反弹；盾反不能反弹盾反 |
| **信息封锁 vs 线索嗅探** | 目标方已施放信息封锁时，线索嗅探无效，但能量仍消耗 |
| **孤注一掷 / 精准换牌后** | pokersolver 自动重新计算最优 5 张，无需手动处理 |
| **能量恰好为 0** | 技能按钮置灰，无其他影响 |
| **玩家超时未行动** | 有注额时自动 fold；无注额时自动 check |
| **玩家断线** | 保留状态 30 秒；超时自动 fold/视为认负并写入战绩 |
| **All-in 边池** | 多人局需计算边池（Side Pot）；All-in 玩家只能赢取其跟注范围内的底池，超出部分在其余玩家间结算 |
| **技能选择超时** | 30 秒未确认则随机分配 2 个技能 |
| **房间创建者离开** | 若对局未开始，房间解散；对局中按断线处理，剩余玩家继续对局 |
| **同时发起翻倍协议** | 顺序触发机制保证不会同时发起；行动顺序靠前的玩家先触发 |
